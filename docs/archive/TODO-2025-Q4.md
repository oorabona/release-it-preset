## Phase 3: Scripts Architecture Refactoring

**Goal:** Apply SOLID principles to `scripts/` and `scripts/lib/` for better maintainability and testability.

### 3.1 Refactor `scripts/lib/` for Single Responsibility

**Current issues:**
- `git-utils.ts` mixes multiple responsibilities (git commands, changelog parsing, file I/O)
- `commit-parser.ts` could be split into parsing logic vs. categorization logic
- Some functions have side effects (console.log) mixed with business logic

**Tasks:**
- [ ] **Split `git-utils.ts` into smaller modules:**
  - `git-utils.ts` - Pure git command execution (getLatestTag, getCurrentBranch, etc.)
  - `changelog-parser.ts` - Parse CHANGELOG.md format
  - `changelog-writer.ts` - Write/update CHANGELOG.md
  - `file-utils.ts` - Generic file operations (if needed)

- [ ] **Separate concerns in `commit-parser.ts`:**
  - `commit-parser.ts` - Parse conventional commit format
  - `commit-categorizer.ts` - Map commit types to changelog categories
  - `commit-formatter.ts` - Format commits for changelog output

- [ ] **Remove side effects from utility functions:**
  - Extract console.log/console.error to caller functions
  - Return error objects instead of logging directly
  - Make functions pure where possible

**Benefits:**
- Easier unit testing (no mocking of file I/O in every test)
- Clearer separation of concerns
- Reusable components
- Better adherence to Single Responsibility Principle

**Estimated effort:** 2-3 days

---

### 3.2 Dependency Inversion in Scripts

**Current issues:**
- Scripts directly import and call git-utils functions
- Hard to test without executing real git commands
- Tight coupling between scripts and utilities

**Tasks:**
- [ ] **Create abstraction interfaces:**
  ```typescript
  // scripts/lib/interfaces/git-provider.ts
  export interface GitProvider {
    getLatestTag(): Promise<string>;
    getCurrentBranch(): Promise<string>;
    getCommitsSinceTag(tag: string): Promise<Commit[]>;
    // ...
  }

  // scripts/lib/implementations/git-provider-impl.ts
  export class GitProviderImpl implements GitProvider {
    // Real git command execution
  }

  // tests/mocks/git-provider-mock.ts
  export class GitProviderMock implements GitProvider {
    // Mock data for testing
  }
  ```

- [ ] **Inject dependencies in main scripts:**
  ```typescript
  // scripts/populate-unreleased-changelog.ts
  export async function populateChangelog(
    gitProvider: GitProvider,
    changelogWriter: ChangelogWriter
  ): Promise<void> {
    // Business logic using injected dependencies
  }

  // CLI entry point
  if (import.meta.main) {
    const gitProvider = new GitProviderImpl();
    const changelogWriter = new ChangelogWriterImpl();
    await populateChangelog(gitProvider, changelogWriter);
  }
  ```

- [ ] **Update tests to use mocks:**
  - Remove file system setup in tests
  - Use in-memory mocks for git/file operations
  - Faster, more reliable tests

**Benefits:**
- Testable without real git repositories
- Easier to maintain and extend
- Follows Dependency Inversion Principle
- Can swap implementations (e.g., GitLab vs GitHub)

**Estimated effort:** 3-4 days

### 3.3 Extract Common Patterns

**Tasks:**
- [ ] **Create base script class/interface:**
  ```typescript
  // scripts/lib/base-script.ts
  export abstract class BaseScript {
    protected abstract validate(): Promise<void>;
    protected abstract execute(): Promise<void>;

    async run(): Promise<void> {
      try {
        await this.validate();
        await this.execute();
      } catch (error) {
        this.handleError(error);
        process.exit(1);
      }
    }

    protected handleError(error: Error): void {
      console.error(`❌ ${error.message}`);
    }
  }
  ```

- [ ] **Standardize error handling:**
  - Create custom error classes (ValidationError, GitError, ChangelogError)
  - Consistent error messages across all scripts
  - Proper exit codes (0 = success, 1 = error, 2 = validation failure)

- [ ] **Extract CLI argument parsing:**
  - Create shared argument parser utility
  - Validate and sanitize arguments once
  - Type-safe argument handling

**Benefits:**
- Consistent behavior across all scripts
- DRY principle (no repeated error handling)
- Easier to add new scripts following the pattern

**Estimated effort:** 1-2 days

### 4.3 Add End-to-End Workflow Tests

**Goal:** Validate complete release workflows from start to finish.

**Tasks:**
- [ ] **Create `tests/e2e/release-workflow.test.ts`:**
  - Test full release cycle: populate → validate → release
  - Test hotfix workflow
  - Test manual-changelog workflow
  - Test retry-publish workflow

  - Verify CHANGELOG.md updates correctly
  - Verify git tags are created

**Example:**
```typescript
describe('E2E: Complete release workflow', () => {
  it('should execute full default release', async () => {
    const repo = await createTempGitRepo();
    await commitChanges(repo, 'feat: new feature');

    // Populate changelog
    await runScript('populate-unreleased-changelog', repo);
    const changelog = readFile(repo, 'CHANGELOG.md');
    expect(changelog).toContain('### Added');
    expect(changelog).toContain('new feature');

    // Validate release
    const validation = await runScript('validate-release', repo);
    expect(validation.exitCode).toBe(0);

    // Simulate release (dry-run)
    const release = await runCLI('default --dry-run', repo);
    expect(release.exitCode).toBe(0);
  });
});
```

**Benefits:**
- Confidence that workflows actually work end-to-end
- Catch integration issues between scripts
- Realistic testing scenarios
